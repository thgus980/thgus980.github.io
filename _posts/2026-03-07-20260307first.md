## AWS Certified Solutions Architect – Associate (SAA)

### Kinesis
- Q. [상황] 전 세계 수천만 명의 사용자가 이용하는 모바일 게임의 실시간 로그 분석 시스템을 설계하고 있습니다. 게임 내에서 발생하는 클릭 이벤트, 아이템 구매, 오류 로그 등 초당 수백만 건의 데이터를 수집하여, 이를 실시간으로 분석 서버와 대시보드 서버, 그리고 장기 보관용 S3 버킷으로 동시에 보내야 합니다.
- [요구사항] 초당 수백만 건의 대규모 데이터를 실시간으로 수집할 수 있어야 함. 데이터를 수집한 후, 여러 소비 주체(분석 엔진, 대시보드, S3 등)가 동일한 데이터를 각자의 속도에 맞춰 독립적으로 처리할 수 있어야 함. 데이터 처리 순서가 엄격하게 보장되어야 함.
- [가장 적절한 서비스는 무엇입니까?]
- A. Amazon Kinesis Data Streams

### 참고
1. SQS vs Kinesis 결정적 차이 (소비자 수) -> SQS: 한 소비자가 메시지를 가져가면, 다른 소비자는 그 메시지를 볼 수 없음 (1대 1 처리). Kinesis: 동일한 데이터 스트림을 여러 소비자(Application)가 동시에, 각각 처음부터 끝까지 읽어갈 수 있습니다. 문제에서 '분석 서버', '대시보드', 'S3'가 모두 같은 데이터를 필요로 하므로 Kinesis가 정답.
2. 데이터의 양과 처리 방식 -> SQS는 주로 개별 작업(주문, 메일 발송 등)을 비동기로 처리할 때 사용. Kinesis는 '스트리밍' 데이터에 최적화되어 있어 초당 수백 메가바이트의 데이터를 쏟아부어도 **샤드(Shard)** 라는 단위로 확장하여 모두 받아낼 수 있음.
3. SQS: 메시지를 한 곳에서 가져가면 사라지기 때문에 여러 곳에서 동시에 분석하기 어렵고, 초당 수백만 건의 스트리밍 데이터를 처리하기에는 비용과 구조 면에서 효율적이지 않음.
4. SNS: 여러 곳에 뿌려줄 순 있지만(Fan-out), 데이터를 일정 기간 보관하거나 소비자가 본인의 속도에 맞춰 과거 데이터를 다시 읽는 기능은 없음.

### 정리
1. **"메시지"** 를 한 번만 처리하고 싶다 → SQS
2. **"데이터 스트림"** 을 여러 곳에서 실시간으로 분석하고 싶다 → Kinesis
3. **"순서 보장"** 이 매우 중요하다 → Kinesis (또는 SQS FIFO)
4. SQS는 큐 서비스, Kinesis는 스트림 서비스(즉, 데이터를 읽어가도 데이터가 사라지지 않고 일정 기간 유지)
