## AWS Certified Solutions Architect – Associate (SAA)

### Interface Endpoint
- Q. 프라이빗 서브넷에 있는 EC2 인스턴스에서 실행되는 애플리케이션이 **Amazon SNS(Simple Notification Service)** 로 메시지를 게시하도록 설정하려고 합니다. 회사의 보안 정책상 이 트래픽은 절대 공용 인터넷을 거쳐서는 안 됩니다. 또한, 이 통신에 대해 특정 **보안 그룹(Security Group)** 을 적용하여 세밀하게 제어하고 싶습니다. 어떤 조치를 취해야 합니까?
  1. SNS용 게이트웨이 VPC 엔드포인트를 생성한다.
  2. SNS용 인터페이스 VPC 엔드포인트를 생성하고 보안 그룹을 할당한다.
  3. 프라이빗 서브넷에 인터넷 게이트웨이(IGW)를 직접 연결한다.
  4. NAT 게이트웨이를 통해 SNS 엔드포인트로 트래픽을 보낸다.
- A. 정답 2번
  - 이유: S3와 DynamoDB를 제외한 대부분의 서비스는 인터페이스 엔드포인트를 사용해야 함. 인터페이스 타입은 서브넷 안에 '가상 랜카드(ENI)'를 꽂는 방식이라 보안 그룹을 직접 걸어서 "어떤 EC2만 이 엔드포인트에 접속할 수 있는지"를 아주 세밀하게 막을 수 있음.
  - 1번은 SNS에서 지원하지 않는 방식(S3/DynamoDB 전용)이고, 4번 NAT 게이트웨이는 결국 인터넷망을 거쳐서 나가기 때문에 "인터넷을 거치지 마라"는 보안 정책에 어긋남.
  - (참고) S3나 DynamoDB인데 돈 이야기가 나오면? -> 게이트웨이 엔드포인트, 그 외 서비스이거나 보안 그룹으로 더 빡빡하게 막고 싶다면? -> 인터페이스 엔드포인트

### Amazon SQS(Simple Queue Service)
- Q. [상황] 사용자가 업로드한 이미지를 분석하여 태그를 다는 서비스를 운영하고 있습니다. 현재는 사용자가 이미지를 올리면 FastAPI 서버가 즉시 분석 작업을 시작하는데, 트래픽이 몰릴 때 서버의 CPU 사용량이 급증하면서 다른 사용자들의 요청에 응답하지 못하는 현상이 발생하고 있습니다. 분석 작업은 시간이 다소 걸려도 상관없지만, 사용자의 업로드 요청은 지연 없이 즉시 완료되어야 합니다.
- [요구사항] 1. 업로드 처리와 이미지 분석 작업을 분리(Decoupling)하여 시스템의 안정성을 높여야 함. 2. 분석 서버가 일시적으로 중단되더라도 데이터(이미지 분석 요청)가 사라지지 않아야 함. 3. 갑작스러운 트래픽 증가에도 시스템 전체가 마비되지 않도록 완충 장치가 필요함.
- 위의 상황에서 가장 적절한 아키텍처 설계는?
  1. 분석 서버의 사양을 더 높은 인스턴스(Scale-up)로 변경한다.
  2. 웹 서버와 분석 서버 사이에 Amazon SQS 대기열을 배치한다.
  3. 분석 작업을 AWS Lambda로 옮겨서 서버리스로 실행한다.
  4. 분석 서버 앞에 로드 밸런서(ALB)를 추가하여 부하를 분산한다.
- A. 정답 2번
  - 디커플링(Decoupling, 결합도 낮추기): SQS는 '메시지 보관함' 역할을 함. 웹 서버는 이미지를 받으면 "분석해줘!"라는 메시지를 SQS에 던지고 바로 사용자에게 "완료!"라고 응답.
  - 분석 서버는 본인의 속도에 맞춰 SQS에서 메시지를 하나씩 꺼내와서 처리. 두 서버가 직접 연결되어 있지 않기 때문에 한쪽이 바빠도 다른 쪽이 멈추지 않음.
  - 내구성 및 완충 작용: 트래픽이 폭주해서 메시지가 쌓여도 SQS가 안전하게 보관해 줌. 분석 서버가 잠깐 죽었다 살아나도, SQS에 쌓여있던 메시지부터 다시 처리를 시작하면 되므로 데이터 유실이 없음.
  - 1번 보기의 경우 비용은 많이 들지만, 여전히 직접 연결된 구조라 트래픽 폭주 시 또 죽을 수 있음.
  - 4번 보기의 경우 로드 밸런서는 실시간 응답을 위한 것이지, 시간이 걸리는 작업을 뒤로 미루는 '완충' 역할에는 적합하지 않음.
  - **애플리케이션을 분리(Decouple)하고 싶다** 거나 **트래픽 급증(Spike)** 에 대비하고 싶다? -> SQS가 정답.
    - 표준 큐(Standard Queue): 처리 속도가 매우 빠르지만 순서가 바뀔 수 있음.
    - FIFO 큐(Frist-In-Frist-Out): 보낸 순서대로 정확히 처리해야 할 때 사용. (예: 결제 처리)
   
### Amazon SNS(Simple Notification Service) + SQS(Simple Queue Service)
- Q. [상황] 온라인 쇼핑몰의 백엔드 시스템을 설계하고 있습니다. 사용자가 '주문 완료' 버튼을 누르면 다음과 같은 세 가지 작업이 동시에 일어나야 합니다.
  1. 배송 시스템에 주문 정보를 전달해야 함.
  2. 인벤토리 시스템에 재고 차감 정보를 전달해야 함.
  3. 마케팅 시스템에 구매 데이터 분석 요청을 전달해야 함.  
  각 시스템은 서로 독립적으로 운영되며, 향후 새로운 시스템(예: 포인트 적립 시스템)이 추가될 수도 있습니다.
- [요구사항] 1. 주문 서버가 각 시스템에 일일이 데이터를 보낼 필요 없이, 한 번의 작업으로 모든 곳에 소식을 알려야 함. 2. 특정 시스템이 점검 중이라도 주문 메시지가 사라지지 않고 나중에 처리되어야 함.
- 위의 상황에서 가장 적절한 아키텍처 설계는?
  1. 주문 서버가 세 개의 각 시스템 API를 직접 호출한다.
  2. 주문 서버가 메시지를 **SNS 주제(Topic)** 에 게시하고, 각 시스템은 이 주제를 구독한다.
  3. 주문 서버가 SNS 주제에 메시지를 게시하고, 각 시스템(배송, 인벤토리, 마케팅) 전용 SQS 대기열을 SNS 주제에 구독시킨다.
  4. 주문 서버가 하나의 큰 SQS 대기열에 메시지를 넣고, 세 시스템이 이를 공유해서 가져간다.
- A. 정답 3번
  - SNS의 역할 (발행/구독 모델): SNS는 '방송국'과 같음. 주문 서버가 "주문 발생!"이라는 메시지를 SNS에 딱 한 번만 던지면(Publish), 이 소식을 듣고 싶어 하는(Subscribe) 수많은 곳에 동시에 뿌려줌. 이를 **'팬아웃(Fan-out)'** 이라고 부름.
  - SNS + SQS 조합의 이유 (안정성): 만약 SNS에서 배송 시스템으로 메시지를 직접 쐈는데, 배송 시스템 서버가 잠시 꺼져 있다면? 메시지는 그대로 증발 -> 그래서 **각 시스템 앞에 SQS(대기열)** 를 하나씩 둠.
    - SNS가 메시지를 3개의 SQS에 각각 복사해서 넣어줌.
    - 어느 한 시스템이 죽어도 해당 시스템의 SQS에 메시지가 안전하게 쌓여 있으므로, 나중에 서버가 살아났을 때 이어서 처리할 수 있음.
  - 1번 보기의 경우 시스템이 추가될 때마다 주문 서버 코드를 계속 수정해야 하므로 확징이 어려움.
  - 2번 보기의 경우 메시지를 즉시 받지 못하면 사라질 위험이 있음.
  - 4번 보기의 경우 하나의 큐를 여러 시스템이 공유하면, 배송 시스템이 가져간 메시지를 인벤토리 시스템은 못 보게 되는 문제가 발생함. **(SQS는 메시지를 한번 가져가면 다른 곳에서 못 보게 숨기기 때문)**
  - **하나의 이벤트를 여러 곳에 전달하고 싶다** -> SNS가 정답, 그리고 그 뒤에 **안전하게(내구성 있게)** 라는 요구 사항이 붙으면 SNS+SQS 조합을 고르자!
  - SNS는 1대 N 방송 (Push 방식), SQS는 1대 1 보관함 (Pull 방식)
