## AWS Certified Solutions Architect – Associate (SAA)

### Storage
- Q. 한 미디어 회사가 전 세계 사용자로부터 대량의 이미지 파일을 업로드받아 저장하는 애플리케이션을 구축하고 있습니다. 이 회사는 다음과 같은 요구 사항을 가지고 있습니다.  
  1. 저장 공간에 제한이 없어야 하며, 수백 테라바이트(TB) 이상의 데이터를 수용해야 함.
  2. 데이터의 유실 없이 매우 높은 내구성(Durability)을 보장해야 함.
  3. 전 세계 어디서든 짧은 지연 시간(Latency)으로 이미지를 다운로드할 수 있어야 함.  
- **최소한의 관리 노력으로 이 요구 사항을 충족하는 가장 적절한 서비스 조합은 무엇입니까?**  
  1. EC2 인스턴스에 대용량 EBS(Elastic Block Store) 볼륨을 연결하여 저장한다.
  2. Amazon S3에 이미지를 저장하고, Amazon CloudFront를 연동하여 배포한다.
  3. **Amazon EFS(Elastic File System)** 를 생성하고 모든 웹 서버에 마운트하여 공유한다.
  4. Amazon RDS의 BLOB 타입을 활용하여 데이터베이스 내부에 직접 저장한다.
- A. 정답 2번
  - Amazon S3: '무제한 저장 공간'과 '99.999999999%(11개의 9)의 내구성'을 자랑하는 객체 스토리지. 이미지 저장에 가장 최적화되어 있음.
  - Amazon CloudFront: 전 세계에 퍼져 있는 거점(Edge Location)을 통해 사용자에게 가장 가까운 곳에서 파일을 전달. 덕분에 지연 시간을 획기적으로 줄여줌.
  - 오답 체크: 1번(EBS)은 저장 용량에 한계가 있고 특정 서버에 종속적. 3번(EFS)은 파일 공유 시스템으로 고성능이지만 비용이 비싸고 전 세계 배포 용도는 아님. 4번(RDS)에 이미지를 직접 저장하는 것은 데이터베이스 성능을 저하시키는 아주 안 좋은 설계.

### EBS(Elastic Block Store)
- 컴퓨터의 SSD/HDD 같은 개념(한 대의 서버에 연결)
- HDD (Hard Disk Drive): 물리적으로 돌아가기 때문에 속도가 상대적으로 느림, 가격 저렴, 대용량 데이터를 저장할 때 유리
- SSD (Solid State Drive): 움직이는 부품이 없어 압도적으로 빠름, 가격 비쌈
- 성능이 필요한 곳엔 SSD를, 저장 공간이 필요한 곳엔 HDD를 섞어서 비용을 최적화하자!

### S3
- 구글 드라이브나 클라우드 같은 개념 (인터넷을 통해 어디서나 접근, 무제한)
- 99.999999999%(11개의 9)의 내구성? 9가 11개나 붙어있는, **99.999999999%** 의 내구성이란, 천만 개의 파일 중 단 1개의 파일을 잃어버리는 데 걸리는 시간이 통계적으로 10,000년이라는 뜻. 사실상 "우리 서비스에 저장한 데이터는 지구가 멸망하기 전까지는 절대 안 사라진다"라고 자신 있게 말하는 것과 같음.
- 내구성(Durability): 데이터가 사라지지 않고 잘 보관되는가? (S3 표준 모델 기준 99.999999999%)
- 가동률(Availability): 지금 당장 파일에 접근(다운로드) 가능한가? (S3 표준 모델 기준 99.99%)
- 내구성 vs 가동률? "데이터 센터 전체가 마비되어 잠시 다운로드가 안 될 순 있어도(가동률 이슈), 데이터 자체가 지워지지는 않는다(내구성)

### CloudFront
- 전 세계에 설치된 캐시 서버 (속도 향상의 핵심)
