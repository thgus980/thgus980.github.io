## AI

### 온톨로지, RAG
- 온톨로지? 단순한 키워드 저장이 아니라 지식 그래프 형태로 데이터를 엮어두는 설계도
- 온톨로지 + RAG (GraphRAG): 기존 RAG는 사용자의 질문과 문장 유사도가 높은 문서를 찾아옴. 하지만 온톨로지를 결합하면 맥락과 관계를 추론할 수 있게 됨. 이것을 요즘 GraphRAG라고 부름.
  - 기존 RAG (Vector Search):
    - "김철수", "관리", "시스템", "서버 끄는 법"이라는 단어가 들어간 문서를 찾음.
    - 만약 문서에 "김철수" 이름이 없고 "IT개발팀 담당자가 관리함"이라고만 써있으면, 김철수가 담당자인지 몰라서 답을 못 찾을 수도 있음.
  - 온톨로지 기반 RAG:
    - 관계 추론: 온톨로지 지도를 봄. [김철수] -> (소속은) -> [IT개발팀] -> (담당 시스템은) -> [ERP]라는 연결 고리를 찾음.
    - 질문 확장: "아, 사용자가 묻는 건 'ERP 시스템' 서버 끄는 법이구나!"라고 파악.
    - 정확한 검색: 매뉴얼에서 'ERP 서버 종료' 부분을 찾아 답변.
- 온톨로지는 RAG 챗봇에게 단어의 뜻과 관계를 알려주는 역할을 함. 이를 통해 추론이 가능해짐.

### LLM(내재된 지식) vs. 온톨로지(외부 지식 그래프)
- LLM: 확률적 감, 유연함, 가르쳐주지 않아도 문맥으로 대충 때려 맞춤, 가끔 엉뚱한 관계를 지어냄
- 온톨로지: Fact 기반, 정확함, 할루시네이션 없음, 구축하기가 번거롭고 유연성 떨어짐
- LLM이 하는 것과 우리가 시스템으로 구축하는 온톨로지는 **확률과 규칙**에 결정적인 차이가 있음

### DB 테이블 RAG화
- 문제점: 숫자와 연산 불가, 정확한 필터링 불가
- 장점: 테이블 안에 긴 줄글 컬럼이 있다면 그 부분에 RAG를 부분 적용하기 -> 이런 텍스트 컬럼만 뽑아서 벡터화하고, 나머지 ID나 날짜 같은 건 '메타데이터'로 붙여서 활용
- DB 테이블 전체를 무작정 RAG로 만드는 건 비효율적 -> 텍스트가 많은 컬럼은 RAG, 숫자가 많고 정확해야 하는 데이터는 MCP 툴 기반으로 처리
